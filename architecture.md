# Технологии
Обязательные технологии: **Python 3.11+** и **PostgreSQL**

Парсинг можно оставить примерно такой же, как и в скрипте: **Selenium** с обходами анти-бот защиты

Из-за того, что каждый запрос выполняется достаточно долго, можно поднять на основе скрипта полноценный сервис,
например на **FastAPI**, который позволяет выполнять запросы асинхронно, и добавить к нему очередь сообщений
(**RabbitMQ**, **Kafka** или **Redis**), чтобы хранить отложенные запросы при большой загрузке

При большом количестве запросов также будет обязателен прокси

Telegram-бот можно встроить в основной сервис с парсингом, но лучше будет вынести его в отдельный сервис. Для разработки
можно использовать библиотеку **Aiogram3**

# Обеспечение устойчивости
Большую часть решений по обеспечению устойчивости я описал в README, здесь добавлю, что обязателен прокси-пул с
ротацией и периодической проверкой на возврат капчи или HTTP-ошибки. Если такая ситуация возникает, нужно останавливать
отправку запросов с этого IP на некоторое время

# Архитектура
Пользователь отправляет запрос в сервис парсинга, в нём происходит решение по началу выполнения сейчас или добавлении в
в очередь сообщений при большой загрузке сервиса

Основной сервис в свою очередь общается с базой данных и сервисов Telegram-бота для отправки уведомлений

# Основные риски
* Блокировка IP - остановка парсинга с заблокированного IP на некоторое время
* Изменение API Озона - уведомление разработчикам о массовом падении сервиса, автоматическое тестирование на тестовых запросах раз в сутки
* Сбой базы данных - создание резервных копий базы данных
* Перегрузка сервиса парсера - увеличение количества экземпляров сервиса при повышении количество запросов

# Бюджет
Если разворачивать базу данных, брокер сообщений и сервисы во внутренней сети, то из можно считать условно бесплатными

Обязательной тратой будет прокси. 50 резидентских IP стоят 100-150 долларов в месяц (у проверенных хостингов), это и
и будет приблизительным месячным бюджетом разворачивания сервиса
